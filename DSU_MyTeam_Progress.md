## 📋 1. 프로젝트 기본 기조 및 아키텍처 정보
*   **프로젝트명**: DSU My_team (동서대학교 MYDEX 팀 매칭 시스템)
*   **개발 기조**: 하나의 `web/index.html` 파일 내부에서 자바스크립트를 이용해 페이지 새로고침 없이 즉각 화면을 전환하는 **단일 페이지 애플리케이션(SPA)** 방식으로 웹 구축.
*   **디자인 규격**: TailwindCSS 기반의 **Pastel Ethereal 테마**, 폰트는 **Plus Jakarta Sans** 사용.
*   **데이터 연동 구조**: 웹 화면(Vanilla JS) ↔ 내장 API 서버(aiohttp, Port 8080) ↔ 로컬 DB(SQLite) + 클라우드 DB(Google Sheets).

### 📅 2026-06-05 (Claude Sonnet 4.6 Thinking - 긴급 버그 4건 수정 세션)

*   **[Bug 2 수정 - 프로그램 선택 데이터 바인딩 오류]** `web/index.html`

*   **[Bug 1 수정 - 메인 홈 UI 누락 복구]** `web/index.html`
    *   **원인**: 대시보드 우측 사이드바의 Discord 카드 아래에 있어야 할 '팀 신청 목록(매칭 대기 중인 팀)으로 이동' 카드 UI 자체가 코드에서 증발해 있었음.

### 📅 2026-06-08 (Antigravity - DB & 매칭 시스템 안정화 세션)
*   **[데이터베이스 구조 안정화 (고유 식별자 추가)]**
    *   **이슈**: 동일한 디스코드 계정으로 여러 번의 웹 신청 테스트 시, 백엔드 및 DB가 디스코드 아이디(`discord_id`)를 기준으로 사용자를 식별하여 신청 데이터가 덮어씌워지거나 오작동하는 문제 발생.
    *   **해결**: SQLite `applications` 테이블에 `unique_id` 컬럼을 신설하여, 디스코드 아이디와 상관없이 웹에서 발급된 고유 아이디를 1순위 식별자로 사용하도록 `db_manager.py` 구조 전면 개편.
*   **[학번 미기재 버그 (구글 시트 검색 로직 역전)]**
    *   **이슈**: 구글 시트에서 학번 데이터를 가져올 때, 회원가입 시 생성된 '학번 없는 빈 행'을 먼저 읽고 검색을 종료해버려 "미기재"로 표출되던 버그.
    *   **해결**: `gsheet_service.py`의 `get_fallback_data` 내 시트 행 검색 순서를 역순(`reversed()`)으로 변경, 가장 최근에 작성된(학번이 포함된) 신청 데이터 행을 최우선으로 읽어오도록 수정.
*   **[유령 대기자 삭제 및 팀 승인 충돌 방지]**
    *   **이슈**: 구글 시트만 초기화하고 로컬 DB를 비우지 않아 "이름미상", "test" 등의 유령 데이터가 계속 대기자 목록에 표시됨. 또한, 동일 디스코드 계정 테스트로 인해 "나는대머리다2,3,4" 승인 시 "이미 매칭 완료된 사용자" 오류 발생.
    *   **해결**: 로컬 DB(SQLite)에서 불필요한 유령 대기자 15명 일괄 삭제 처리. 팀장 대시보드([가입 승인]) 로직을 `discord_id` 대신 `unique_id` 기준으로 승인 및 조회(`get_application_by_uid`)하도록 `api/routes.py`와 `web/index.html` 완전 연동. 이로써 동일 디스코드 계정으로 100명을 번갈아 테스트해도 가입 승인 충돌 제로화 달성.
